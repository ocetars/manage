**为了解决“文档同步”问题，强烈建议合并到一个仓库（Monorepo）。**

如果目标是治理好一个 15K Star 级别的项目，**“代码与文档分离”通常是导致文档腐烂的头号杀手**。

以下是深度的利弊分析和解决方案：

---

### 1. 为什么要合并？（The Case for Monorepo）

目前业界的主流开源框架（如 FastAPI, React, Vue, LangChain）几乎都将文档放在主仓库的 `docs/` 或 `website/` 目录下。

#### 核心优势：原子化提交 (Atomic Commits)
这是解决“不同步”的终极法宝。
*   **分仓痛点：** 开发者修改了 API -> 提交代码 PR -> **（忘记去另一个仓库提 PR）** -> 代码合并了 -> 文档没变 -> 用户骂街。
*   **合仓优势：** 开发者修改了 API -> 必须在**同一个 PR** 里修改 `docs/` 下的文档 -> Reviewer 看到文档没改拒绝合并 -> **强一致性**。

---

### 3. 最佳实践推荐：迁移合并方案

针对 AstrBot，我建议采用 **“源码主导，文档内嵌”** 的结构。

即使你有独立的文档网站（比如部署在 Vercel/Netlify），它完全可以由主仓库的一个子目录构建并部署。

**建议的新目录结构：**

```text
AstrBot/
├── .github/              # CI/CD workflows
├── astrbot/              # Python 源代码 (Core)
├── tests/                # 测试代码
├── docs/                 # 文档源文件 (Markdown)
│   ├── en/
│   ├── zh/
│   └── api/              # 这里由 mkdocstrings 自动生成
├── website/              # 文档网站的工程配置 (如果是 Docusaurus/VitePress)
│   ├── package.json
│   └── docusaurus.config.js # 配置指向上一级的 ../docs
├── pyproject.toml
└── README.md
```

**迁移步骤：**

1.  **搬家：** 将现有的文档仓库内容搬运到主仓库的 `docs/` 或 `website/` 目录。
2.  **配置部署：** 修改 Vercel/Netlify/GitHub Pages 的设置，将 `Root Directory` 指向 `website/` (或者构建脚本所在目录)。
3.  **配置 CI：**
    *   设置 GitHub Action，只有当 `docs/` 或 `website/` 目录发生变动时，才触发网站构建（节省资源）。
    *   设置 CI 规则：代码变更检测到 API 签名变化时，检查 `docs/` 是否有变动（可选）。

### 4. 结论

**“有独立网站”不代表“要有独立仓库”。**

Vue.js, React, Next.js 都有非常炫酷的独立文档网站，但它们的文档源码全都在主仓库里。

**给 AstrBot 的最终建议：**

为了长治久安，为了让“代码变动自动更新文档”的技术方案（mkdocstrings）能落地，**请把文档仓库合并进主仓库**。
